## 2. 单周期指令执行流程

模拟器采用严格的单周期功能模型——**没有流水线，一条指令在一个 `sim_step()` 调用内完整执行**。

`sim_step()` 内部流程（`src/src/simulator.c`）：

```
sim_step(sim):
  ① fetch:     mmu_read_32(pc) → instr
  ② decode:    cpu_decode(instr) → DecodedInstr d
  ③ execute:   cpu_execute(sim, &d, &next_pc)
               — 写寄存器、可能修改 next_pc
  ④ writeback: sim->cpu.pc = next_pc
               — 默认 pc+4，分支/跳转覆盖
  ⑤ x0 硬连线: sim->cpu.regs[0] = 0
  ⑥ 统计:      instr_count++, cycle_count++
```

`simulator.h` 注释确认：*"执行单条指令（取指 → 译码 → 执行 → 写回）"* — 每条指令严格按照取指→译码→执行→写回的顺序进行。`cycle_count` 等于 `instr_count`（每条指令恰好一个周期），这是单周期模型的典型特征。

## 3. CPU 状态

定义于 `src/include/cpu/cpu.h`。

### 3.1 通用寄存器

- `uint32_t regs[32]` — x0–x31。x0 硬连线为 0（每条指令执行后强制执行）
- `uint32_t fregs[32]` — f0–f31 浮点寄存器（F 扩展）

### 3.2 程序计数器

- `uint32_t pc` — 当前正在执行的指令地址

### 3.3 控制标志

- `bool running` — CPU 执行标志（false = 停止）
- `PrivilegeLevel priv` — 当前特权级（0=User, 1=Supervisor, 3=Machine）。初始化为 `PRIV_MACHINE`

### 3.4 CSR 寄存器（Machine 模式）

| 寄存器 | 位宽 | 说明 |
|--------|------|------|
| `mstatus` | 32 | 全局中断使能 (MIE[3])、异常前中断使能 (MPIE[7])、异常前特权级 (MPP[12:11]) |
| `mtvec` | 32 | 陷阱向量基址。BASE[31:2] 为处理程序入口，MODE[1:0] 为模式（0=直接，1=向量） |
| `mepc` | 32 | 异常指令地址——异常发生时保存触发异常的指令 PC |
| `mcause` | 32 | 陷阱原因。bit[31]=中断标志，bits[30:0]=异常编号（见 `ExceptionType` 枚举） |
| `mtval` | 32 | 陷阱附加信息（页错误时=错误地址，非法指令时=指令编码） |

### 3.5 Simulator 级别附加状态

定义于 `src/include/simulator.h`：

| 字段 | 类型 | 说明 |
|------|------|------|
| `mmu` | `MMUState` | Sv32 页表 + satp 寄存器 |
| `pmem` | `PhysicalMemory` | 128MB 字节数组，支持 8/16/32 位小端访问 |
| `breakpoints` | `Breakpoint*` | 软件断点动态数组 |
| `single_step` | `bool` | 单步执行标志 |
| `debug_mode` | `bool` | 调试器是否接管 |
| `instr_count` / `cycle_count` | `uint64_t` | 性能计数器（单周期中 `cycle_count == instr_count`） |

## 4. 指令译码

定义于 `src/include/cpu/decode.h`，实现在 `src/src/cpu/decode.c`。

### 4.1 位域提取宏

```c
OPCODE(instr)  = instr & 0x7F            // [6:0]
RD(instr)      = (instr >> 7) & 0x1F     // [11:7]
FUNCT3(instr)  = (instr >> 12) & 0x7     // [14:12]
RS1(instr)     = (instr >> 15) & 0x1F    // [19:15]
RS2(instr)     = (instr >> 20) & 0x1F    // [24:20]
FUNCT7(instr)  = (instr >> 25) & 0x7F    // [31:25]
RS3(instr)     = (instr >> 27) & 0x1F    // [31:27] (R4 FMA 格式)
```

### 4.2 立即数提取宏（5 种格式）

| 宏 | 格式 | 位分布 |
|----|------|--------|
| `IMM_I(instr)` | I-type | 12 位，取自 [31:20]，算术右移 20 位完成符号扩展 |
| `IMM_S(instr)` | S-type | 12 位，拆分为 [31:25]（高 7 位）+ [11:7]（低 5 位） |
| `IMM_B(instr)` | B-type | 13 位（bit[0] 恒为 0），分散在 4 段拼合 |
| `IMM_U(instr)` | U-type | 高 20 位取自 [31:12]，低 12 位为 0 |
| `IMM_J(instr)` | J-type | 21 位（bit[0] 恒为 0），分散在 4 段拼合 |

### 4.3 译码结果结构体

```c
typedef struct DecodedInstr {
    uint8_t  opcode;     // [6:0]  指令类型大类
    uint8_t  rd;         // 目标寄存器号 (0-31)
    uint8_t  rs1;        // 源寄存器1号  (0-31)
    uint8_t  rs2;        // 源寄存器2号  (0-31)
    uint8_t  funct3;     // 3 位功能码
    uint8_t  funct7;     // 7 位功能码 (R-type)
    int32_t  imm;        // 立即数（已符号扩展到 32 位）
    uint8_t  rs3;        // 源寄存器3号 (R4 FMA 格式)
} DecodedInstr;
```

### 4.4 `cpu_decode()` 逻辑

1. 提取所有固定位置字段（opcode, rd, funct3, rs1, rs2, funct7, rs3）
2. 按 opcode 选择立即数格式：

| opcode | 格式 | 对应指令 |
|--------|------|----------|
| `0x37`, `0x17` | `IMM_U` | LUI, AUIPC |
| `0x6F` | `IMM_J` | JAL |
| `0x67`, `0x03`, `0x13`, `0x0F`, `0x73`, `0x07` | `IMM_I` | JALR, Load, OP-IMM, FENCE, SYSTEM, FLW |
| `0x63` | `IMM_B` | Branch |
| `0x23`, `0x27` | `IMM_S` | Store, FSW |
| `0x33`, FMA | imm=0 | R-type（无立即数） |
| 其他 | opcode=0 | 非法指令标志 |

### 4.5 `cpu_disasm()`

提供人类可读的反汇编输出（供 Debugger 使用）。使用 37 条指令的查找表，按 `{opcode, funct3, funct7}` 映射到指令名称，然后按指令格式（R/I/S/B/U/J/SYSTEM）格式化输出。

## 5. 指令执行

### 5.1 `cpu_execute()` 分发器

位置：`src/src/cpu/execute/execute.c`

一个 `switch` 语句按 `dec->opcode` 分发。进入 switch 前，`*next_pc` 默认为 `pc + 4`。

| Opcode | 处理函数 | 文件 | 说明 |
|--------|----------|------|------|
| `0x37` | `exec_lui()` | `exec_rv32i.c` | Load upper immediate |
| `0x17` | `exec_auipc()` | `exec_rv32i.c` | Add upper imm to PC |
| `0x13` | `exec_op_imm()` | `exec_rv32i.c` | 9 条 ALU 立即数指令 |
| `0x33` | `exec_op()` | `exec_rv32i.c` | 10 条 ALU 寄存器指令 + M 扩展分流 |
| `0x03` | `exec_load()` | `exec_rv32i.c` | LB / LH / LW / LBU / LHU |
| `0x23` | `exec_store()` | `exec_rv32i.c` | SB / SH / SW |
| `0x63` | `exec_branch()` | `exec_rv32i.c` | BEQ / BNE / BLT / BGE / BLTU / BGEU |
| `0x6F` | `exec_jal()` | `exec_rv32i.c` | Jump and link |
| `0x67` | `exec_jalr()` | `exec_rv32i.c` | Jump and link register |
| `0x73` | `exec_system()` | `exec_rv32i.c` | ECALL / EBREAK / MRET |
| `0x0F` | `exec_fence()` | `exec_rv32i.c` | NOP（单核中无意义） |
| `0x07` | `exec_load_fp()` | `exec_f.c` | 浮点 Load (FLW) |
| `0x27` | `exec_store_fp()` | `exec_f.c` | 浮点 Store (FSW) |
| `0x53` | `exec_fp_op()` | `exec_f.c` | 浮点运算 (FADD.S, FMUL.S 等) |
| `0x43/47/4B/4F` | `exec_fma()` | `exec_f.c` | 融合乘加 (FMA) |
| 其他 | `cpu_trap()` | `execute.c` | 非法指令异常 |

### 5.2 执行约定

每个处理函数内部：

1. 从 `sim->cpu.regs[dec->rs1/rs2]` 读取源操作数
2. 执行计算（ALU、分支比较、地址计算）
3. 结果写入 `sim->cpu.regs[dec->rd]`
4. 分支/跳转指令：覆盖 `*next_pc`（默认值为 `pc+4`）
5. Load/Store 指令：通过 `mmu_read_*/mmu_write_*` 访问内存（内部走地址翻译→物理访问）
6. SYSTEM 指令：`ecall` 触发 `cpu_trap(sim, EXC_ECALL_M, 0)`，然后 `syscall_handler(sim)` 读取 `a7` 分发到 exit/write/read/brk

### 5.3 异常处理 —— `cpu_trap()`

位置：`src/src/cpu/execute/execute.c`

六步 RISC-V 陷阱序列：

```
① mepc    ← pc          （保存异常指令地址）
② mcause  ← exc         （保存异常类型）
③ mtval   ← tval        （保存附加信息）
④ mstatus: MPP ← priv, MPIE ← MIE, MIE ← 0
⑤ priv    ← PRIV_MACHINE（进入机器模式）
⑥ if mtvec != 0: pc ← mtvec & ~0x3（跳转处理程序）
   if mtvec == 0: cpu->running ← false（停机，ecall 特殊处理除外）
```

## 6. 主循环

### 6.1 `main()` —— `src/src/main.c`

1. 解析 CLI 参数：`-s`（调试模式）、`-t`（跟踪，未实现）、`-w <port>`（Web 调试器）、ELF 路径
2. `sim_init(&sim)` — 分配 128MB 物理内存、初始化 MMU、初始化 CPU（寄存器=0、priv=M、pc=0）、分配断点数组（初始容量 16）
3. `sim_load_elf(&sim, elf_path)` — 加载 ELF：验证头、加载 `PT_LOAD` 段（等同映射 paddr=vaddr）、在 `0xC0000000` 设栈、`cpu.pc = entry`、`cpu.regs[REG_SP] = stack_top`
4. 分发到 `sim_run(&sim)` / `debugger_run(&sim)` / `web_server_start(&sim, port)`
5. `sim_destroy(&sim)` — 恢复断点、释放分配

### 6.2 `sim_run()` —— `src/src/simulator.c`

```c
void sim_run(Simulator *sim) {
    sim->cpu.running = true;

    while (sim->cpu.running) {
        sim_step(sim);              // 单周期: fetch → decode → execute → writeback

        if (sim->single_step) {     // 单步模式: 执行一条后暂停
            sim->single_step = false;
            break;
        }
    }

    printf("\nSimulation ended.\n");
    printf("Instructions: %" PRIu64 "  Cycles: %" PRIu64 "\n",
           sim->instr_count, sim->cycle_count);
}
```

### 6.3 `sim_step()` —— `src/src/simulator.c`

执行核心是一条简单的顺序链：

```
Fetch:      mmu_read_32(pc) — 经 MMU（Bare 模式 vaddr=paddr，或 Sv32 翻译）
Decode:     cpu_decode(instr) → DecodedInstr d
Execute:    cpu_execute(sim, &d, &next_pc) — 默认 next_pc=pc+4，分支/跳转覆盖
Writeback:  sim->cpu.pc = next_pc
x0 硬连线:  sim->cpu.regs[0] = 0
统计:       instr_count++, cycle_count++
```

`cycle_count` 与 `instr_count` 每条指令同时递增 — 明确的单周期假设。CPU 在 `cpu_trap()` 将 `running` 设为 `false` 时停止（当 `mtvec == 0` 时），或由 `SYS_exit` 系统调用处理函数设置 `running = false`。

## 7. 流水线相关代码

**本项目当前没有任何流水线代码。** 搜索 "pipeline"、"pipelining"、"stage" 仅找到：

- `src/test/e2e/e2e_test.c` 中的注释：*"Test 1: Full Pipeline (minimal.elf)"* — 此处 "pipeline" 泛指取指→译码→执行这条完整的指令处理链，不是微架构意义上的流水线。

数据结构也全部面向单周期模型：

- `DecodedInstr` 仅存储单条指令的译码字段，没有流水线寄存器概念
- `CPU` 只有一个 `pc`、一组 `regs[32]`，没有流水线级间寄存器、转发路径或冒险检测逻辑
- `sim_step()` 原子性地完成一条指令的取指→译码→执行→写回，指令间没有重叠
- `cycle_count` 每条指令递增 1，典型单周期假设

流水线和多周期模型的设计见 [程序设计2.md](程序设计2.md)。
