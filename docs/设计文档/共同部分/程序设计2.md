# 程序设计（二）—— 多周期 & 流水线 CPU 模型

> 本文档描述 RISC-V 模拟器的多周期与流水线 CPU 控制器设计。单周期模型见 [程序设计.md](程序设计.md)。

---

## 1. 数据通路 / 控制器分离架构

模拟器重构为支持三种 CPU 时序模型，遵循 Patterson & Hennessy 教材设计模式（§4.4–§4.8）。核心原则：**数据通路共享，控制器独立**。

```
                        sim_step()
                     ┌───────────────┐
                     │ switch(model) │
                     │  SINGLE → single_cycle.c
                     │  MULTI  → multi_cycle.c
                     │  PIPE   → pipeline.c
                     └───────┬───────┘
                             │
              ┌──────────────┴──────────────┐
              │     Shared Datapath          │
              │                              │
              │  alu_compute()      纯组合   │
              │  alu_branch_cond()  零依赖   │
              │  alu_select_op()    零副作用 │
              │  cpu_decode()       译码     │
              │  cpu_execute()      执行     │
              │  mmu_read/write_*   访存     │
              │  cpu_trap()         异常     │
              └──────────────────────────────┘
```

`CpuModel` 枚举（`simulator.h`）：

```c
typedef enum {
    MODEL_SINGLE_CYCLE = 0,  // 1 调用 = 1 条完整指令
    MODEL_MULTI_CYCLE  = 1,  // 1 调用 = 1 个 FSM 状态 (CPI > 1)
    MODEL_PIPELINE     = 2,  // 1 调用 = 1 个时钟周期 (5 级重叠)
} CpuModel;
```

## 2. 多周期控制器

> 文件：`src/src/cpu/controller/multi_cycle.c`（~550 行）

### 2.1 设计概述

多周期控制器将指令执行拆分为 5 状态有限状态机。每次调用 `sim_step_multi()` 推进一个 FSM 状态（= 一个时钟周期）。不同指令类型占用不同周期数，实现真实的 CPI > 1。

### 2.2 FSM 状态转移图

```
                     ┌──────────────────────────────────┐
                     │                                  │
                     ▼                                  │
               ┌──────────┐                             │
          ┌───▶│  MC_IF   │────┐                        │
          │    │  Fetch   │    │                        │
          │    └──────────┘    │                        │
          │                    ▼                        │
          │    ┌──────────┐                             │
          │    │  MC_ID   │                             │
          │    │ Decode   │                             │
          │    └──────────┘                             │
          │          │                                  │
          │          ▼                                  │
          │    ┌──────────┐    ┌──────────┐            │
          │    │  MC_EX   │───▶│ MC_MEM   │            │
          │    │ Execute  │    │ Memory   │            │
          │    └──────────┘    └──────────┘            │
          │          │               │                  │
          │          │               ▼                  │
          │          │          ┌──────────┐           │
          │          │          │  MC_WB   │           │
          │          │          │ Writeback│           │
          │          │          └──────────┘           │
          │          │               │                  │
          │          ▼               ▼                  │
          │    ┌─────────────────────────────┐          │
          └────│     Instruction Done         │──────────┘
               └─────────────────────────────┘
```

### 2.3 各指令类型周期数

| 指令类型 | 周期数 | 状态路径 | 示例 |
|---------|:-----:|---------|------|
| R-type ALU | 4 | IF → ID → EX → WB | ADD, SUB, SLL, XOR |
| I-type ALU | 4 | IF → ID → EX → WB | ADDI, ORI, ANDI |
| Load | 5 | IF → ID → EX → MEM → WB | LW, LH, LB, LBU, LHU |
| Store | 4 | IF → ID → EX → MEM | SW, SH, SB |
| Branch | 3 | IF → ID → EX | BEQ, BNE, BLT, BGE |
| Jump (JAL/JALR) | 3 | IF → ID → EX | JAL, JALR |
| LUI / AUIPC | 4 | IF → ID → EX → WB | LUI, AUIPC |
| SYSTEM / FENCE | 3 | IF → ID → EX | ECALL, EBREAK, MRET |
| M 扩展 | 4 | IF → ID → EX → WB | MUL, DIV, REM |
| F 扩展 ALU | 4 | IF → ID → EX → WB | FADD.S, FMUL.S |
| FLW (FP Load) | 5 | IF → ID → EX → MEM → WB | FLW |
| FSW (FP Store) | 4 | IF → ID → EX → MEM | FSW |

### 2.4 中间寄存器（`MultiCycleState`）

定义于 `simulator.h`，保存跨 FSM 状态的中间结果：

```c
typedef struct {
    uint8_t  state;      // 当前 FSM 状态: 0=IF, 1=ID, 2=EX, 3=MEM, 4=WB
    uint32_t ir;         // 指令寄存器（IF 阶段锁存）
    uint32_t pc;         // 当前指令的 PC
    uint32_t a, b;       // 寄存器值 rs1_val, rs2_val（ID 阶段锁存）
    uint32_t alu_out;    // ALU 输出 / 地址计算结果（EX 阶段锁存）
    uint32_t mdr;        // 内存数据寄存器（MEM 阶段锁存）
    uint32_t next_pc;    // 下一条指令的 PC
    uint32_t temp_pc;    // ID 阶段预计算的跳转目标
} MultiCycleState;
```

### 2.5 各状态详细动作

#### MC_IF — 取指

- `IR ← Mem[PC]` — 通过 `mmu_read_32` 读取 32 位指令
- `next_pc ← PC + 4` — 顺序执行的默认值
- 取指异常 → `cpu_trap`，`instr_count++`，重置到 `MC_IF`

#### MC_ID — 译码 + 读寄存器

- `d = cpu_decode(IR)`
- `A ← regs[rs1]`、`B ← regs[rs2]`
- 预计算 B-type 和 J-type 的跳转目标（`temp_pc`）

#### MC_EX — 执行 / 地址计算

```
R-type (0x33):  ALUOut ← A op B（M 扩展走 exec_m_muldiv）
I-type (0x13):  ALUOut ← A op imm
Load/Store:     ALUOut ← A + imm（有效地址）
Branch:         if (alu_branch_cond) next_pc = temp_pc; instr_done
JAL:            regs[rd] ← PC+4; next_pc ← temp_pc; instr_done
JALR:           regs[rd] ← PC+4; next_pc ← (A+imm) & ~1; instr_done
SYSTEM:         ecall / ebreak / mret 处理; instr_done
FENCE:          NOP; instr_done
```

#### MC_MEM — 内存访问

- **Load**：`MDR ← Mem[ALUOut]`，按 `funct3` 分宽度：
  - `LB` — 字节加载 + 符号扩展
  - `LH` — 半字加载 + 符号扩展
  - `LW` — 字加载（不扩展）
  - `LBU` / `LHU` — 无符号加载 + 零扩展
- **Store**：`Mem[ALUOut] ← B`，按 `funct3` 分宽度（SB / SH / SW）
- 访存异常 → `cpu_trap`

#### MC_WB — 写回

- 整数：`regs[rd] ← ALUOut`（Load 指令用 `MDR`）
- 浮点：结果已在 EX 阶段写入 `fregs[rd]`，此处跳过
- `x0` 硬连线为 0
- `instr_count++`、`pc ← next_pc`、`state ← MC_IF`

### 2.6 辅助函数

```c
mc_needs_mem(opcode)          // Load (0x03, 0x07) 和 Store (0x23, 0x27) 返回 true
mc_needs_wb_after_ex(opcode)  // R-type, I-type, LUI, AUIPC, FP 指令返回 true
mc_is_fp_op(opcode)           // FP load/store/op/fma opcode 返回 true
```

## 3. 流水线控制器

> 文件：`src/src/cpu/controller/pipeline.c`（~480 行）

### 3.1 设计概述

实现经典五级 RISC 流水线（Patterson & Hennessy §4.7–§4.8）。每次调用 `sim_step_pipeline()` 推进一个时钟周期。最多 5 条指令同时在 IF → ID → EX → MEM → WB 五级中重叠执行。理想 CPI = 1.0。

```
       ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
Clock  │  IF  │──▶│  ID  │──▶│  EX  │──▶│ MEM  │──▶│  WB  │
Cycle  └──────┘   └──────┘   └──────┘   └──────┘   └──────┘
           ▲           ▲           ▲           ▲
           │           │           │           │
      ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
      │Instr │    │Instr │    │Instr │    │Instr │     Instr i
      │ i+4  │    │ i+3  │    │ i+2  │    │ i+1  │   completes
      └──────┘    └──────┘    └──────┘    └──────┘       │
                                                          ▼
                                                     RegFile
```

### 3.2 流水线寄存器

四级段间寄存器保存各阶段之间的数据流动。全部定义于 `simulator.h`。

```
  IF/ID                  ID/EX                   EX/MEM                MEM/WB
┌──────────┐         ┌────────────┐         ┌──────────────┐      ┌──────────────┐
│ pc       │         │ pc         │         │ pc           │      │ alu_result   │
│ instr    │─DECODE─▶│ d(Decoded) │──ALU──▶│ alu_result   │─MEM─▶│ mem_data     │
│ valid    │         │ rs1_val    │         │ rs2_val      │      │ rd           │
└──────────┘         │ rs2_val    │         │ rd           │      │ reg_write    │
                     │ valid      │         │ opcode       │      │ is_load      │
                     └────────────┘         │ funct3       │      │ is_fp_load   │
                                            │ reg_write    │      │ valid        │
                                            │ mem_read     │      └──────┬───────┘
                                            │ mem_write    │             │
                                            │ is_fp        │        ┌────▼────┐
                                            │ valid        │        │ RegFile │
                                            └──────────────┘        └─────────┘
```

| 寄存器 | 前级→后级 | 关键字段 |
|--------|----------|----------|
| **PipeIFID** | IF → ID | `pc`, `instr`, `valid` |
| **PipeIDEX** | ID → EX | `pc`, `d`（完整 DecodedInstr）, `rs1_val`, `rs2_val`, `valid` |
| **PipeEXMEM** | EX → MEM | `pc`, `alu_result`, `rs2_val`（store 数据）, `rd`, `opcode`, `funct3`, 控制信号（`reg_write`, `mem_read`, `mem_write`, `is_fp`）, `valid` |
| **PipeMEMWB** | MEM → WB | `alu_result`（非 Load 指令直通）, `mem_data`（Load 数据）, `rd`, 控制信号（`reg_write`, `is_load`, `is_fp_load`）, `valid` |

### 3.3 控制器状态（`PipelineState`）

```c
typedef struct {
    PipeIFID   if_id;           // IF/ID 流水线寄存器
    PipeIDEX   id_ex;           // ID/EX 流水线寄存器
    PipeEXMEM  ex_mem;          // EX/MEM 流水线寄存器
    PipeMEMWB  mem_wb;          // MEM/WB 流水线寄存器
    uint64_t   stall_cycles;    // Load-use 冒险停顿计数
    uint64_t   flush_cycles;    // 分支/跳转/陷阱冲刷计数
} PipelineState;
```

### 3.4 逆序处理策略

每个时钟周期按 **逆序**（WB → MEM → EX → ID → IF）处理各阶段。这是正确性的关键：

**WB 先于 ID**：WB 先写寄存器文件，ID 后读寄存器文件——同周期内 WB 写入的值能被 ID 读到，WB→ID 路径无需转发。这模拟了真实硬件中前半周期写、后半周期读的时序。

**EX 使用上一周期的流水线寄存器**：EX 中的转发多路选择器读取的是上一周期的 EX/MEM 和 MEM/WB 寄存器值，当前周期的 MEM 和 WB 还没有更新它们——数据一致性得到保证。

```
sim_step_pipeline() 处理顺序：

① 排空检查       !running && 全部 invalid → return
② 冒险检测       Load-use: ID/EX 是 Load 且 rd 匹配 IF/ID 的 rs
③ WB 阶段        写 mem_wb → regfile / fregs, instr_count++
④ MEM 阶段       Load: 读内存 → next_mem_wb.mem_data
                  Store: 写内存
                  ALU: 直通 alu_result
⑤ EX 阶段        转发 MUX → ALU → next_ex_mem
                  分支决议 → 跳转则 flush
                  陷阱 (ecall/ebreak) → flush
⑥ ID 阶段        译码 if_id → next_id_ex, 读寄存器
⑦ IF 阶段        取指: pc → next_if_id, pc += 4
⑧ stall/flush    stall: 保持 if_id, id_ex 变 NOP
                  覆盖  flush: if_id + id_ex 无效化
⑨ 推进寄存器     if_id/id_ex/ex_mem/mem_wb ← next_*
```

### 3.5 数据冒险 —— Forwarding（前递）

RAW（Read After Write）数据冒险在 EX 阶段通过两条转发路径检测和解决：

```
                  ┌──────────┐
   EX/MEM          │   EX     │          MEM/WB
┌──────────┐       │          │       ┌──────────┐
│alu_result│───────┼──▶MUX◀───┼───────│alu_result│
│  rd      │       │  ▲    ▲  │       │mem_data  │
│reg_write │       │  │    │  │       │  rd      │
└──────────┘       │  │    │  │       │reg_write │
                   │  │    │  │       └──────────┘
                   │  │    │  │
             ID/EX │  │    │  │
          ┌────────┴──┘    └──┴────────┐
          │ rs1_val       rs2_val     │
          │ d.rs1         d.rs2       │
          └───────────────────────────┘
```

**转发优先级**：EX/MEM > MEM/WB（最新结果优先）。

#### EX/MEM 转发（优先级 1）

| 条件 | 转发值 |
|------|--------|
| `EX/MEM.reg_write && rd ≠ 0 && rd == ID/EX.rs1` | `EX/MEM.alu_result` |
| `EX/MEM.reg_write && rd ≠ 0 && rd == ID/EX.rs2` | `EX/MEM.alu_result` |

#### MEM/WB 转发（优先级 2）

| 条件 | 转发值 |
|------|--------|
| 同上条件，且 EX/MEM 未转发该寄存器 | `WB.is_load ? mem_data : alu_result` |

> **FP 转发**：暂未实现。FP 运算在 EX 阶段直接写 `fregs[]` 并设 `reg_write = false`，排除在整数转发之外。FP-FP RAW 冒险在相邻 FP 指令存在时会得到错误结果。

### 3.6 数据冒险 —— Load-Use Stall（停顿）

Load-use 冒险**无法仅靠转发解决**：Load 在 EX 阶段产生结果，到 MEM 阶段结束时才可用；但依赖它的指令在下一个周期（Load 的 MEM 阶段）就需要在 EX 使用该值。必须停顿一个周期。

```
Cycle N:   Load 在 EX, 依赖指令在 ID  → 检测到冒险
Cycle N+1: Load 在 MEM, NOP(气泡) 在 EX, 依赖指令重新译码在 ID
Cycle N+2: Load 在 WB, NOP 在 MEM, 依赖指令在 EX → MEM/WB 转发!
```

**检测逻辑**（每周期在 EX 阶段前执行）：

```c
// ID/EX 是 Load 且其 rd 匹配 IF/ID 的 rs1 或 rs2
bool id_ex_is_load = (p->id_ex.d.opcode == 0x03 || p->id_ex.d.opcode == 0x07);

if (p->id_ex.valid && id_ex_is_load && p->id_ex.d.rd != 0) {
    if (p->if_id.valid) {
        uint8_t rs1 = RS1(p->if_id.instr);
        uint8_t rs2 = RS2(p->if_id.instr);
        if (rs1 == p->id_ex.d.rd || rs2 == p->id_ex.d.rd) {
            stall = true;  // 保持 IF/ID, ID/EX 插入 NOP
        }
    }
}
```

停顿 1 周期后，Load 到达 MEM/WB，转发路径将加载值传给依赖指令的 EX 阶段。

### 3.7 控制冒险 —— 分支预测 + 冲刷

**预测策略**：始终预测 **不跳转**（predict not-taken）。流水线投机执行 `PC+4`、`PC+8`……

**误预测惩罚**：当分支实际跳转时（在 EX 阶段决议），`PC+4`（在 ID/EX）和 `PC+8`（在 IF/ID）处的指令是错误的，必须**冲刷**（无效化）。代价为 2 个周期。

```
Cycle N:   分支在 EX → 条件成立，跳转
            → 冲刷 IF/ID 和 ID/EX（PC+4、PC+8 的指令错误）
            → cpu.pc ← branch_target
Cycle N+1: 分支在 MEM, 气泡在 EX, 气泡在 ID
            → IF 从 branch_target 取指
```

| 指令 | 行为 | 冲刷 |
|------|------|:----:|
| Branch (0x63) | 条件成立时跳转 | ✅ |
| JAL (0x6F) | 无条件跳转 + 链接 | ✅ |
| JALR (0x67) | 无条件跳转 + 链接 | ✅ |
| MRET | PC ← mepc | ✅ |
| ECALL / EBREAK | 陷阱 → 冲刷 IF/ID + ID/EX | ✅ |

### 3.8 异常处理与流水线排空

当 EX 阶段（ecall/ebreak）或 MEM 阶段（load/store fault）发生陷阱时，`cpu_trap()` 将 `cpu->running` 设为 `false`。但陷阱指令**之前**已在 MEM 和 WB 阶段的指令应正常完成。

**三步排空机制**：

1. **立即冲刷** — IF/ID 和 ID/EX 在陷阱发生时立刻无效化（这些是陷阱指令之后取入的，不应执行）
2. **自然排空** — 陷阱指令和之前的指令留在 EX/MEM 和 MEM/WB 中，随后续 1-2 个周期自然流过 MEM 和 WB 阶段
3. **`sim_run` 排空循环** — 主循环退出后，流水线模式额外调用 `sim_step_pipeline()` 直到所有寄存器无效：

```c
if (sim->cpu_model == MODEL_PIPELINE) {
    while (1) {
        bool empty = !sim->pipe.if_id.valid && !sim->pipe.id_ex.valid &&
                     !sim->pipe.ex_mem.valid && !sim->pipe.mem_wb.valid;
        if (empty) break;
        sim_step_pipeline(sim);
    }
}
```

### 3.9 EX 阶段按 opcode 分发表

EX 阶段的 `switch` 语句处理所有 opcode，使用转发后的值（`forward_a`、`forward_b`）代替直接读寄存器：

| Opcode | EX 阶段操作 | 控制信号 |
|--------|------------|----------|
| `0x33` (OP) | `alu_compute(op, fw_a, fw_b)`；M 扩展临时保存/恢复 regs 后调 `exec_m_muldiv` | `reg_write=true` |
| `0x13` (OP-IMM) | `alu_compute(op, fw_a, imm)` | `reg_write=true` |
| `0x03` (LOAD) | `alu_result = fw_a + imm` | `reg_write=true`, `mem_read=true` |
| `0x23` (STORE) | `alu_result = fw_a + imm`, `rs2_val = fw_b` | `mem_write=true` |
| `0x63` (BRANCH) | `alu_branch_cond` → 跳转则 `cpu.pc=target`, `flush=true` | — |
| `0x6F` (JAL) | `alu_result = pc+4`（链接地址）, `cpu.pc = pc+imm`, `flush=true` | `reg_write=true` |
| `0x67` (JALR) | `alu_result = pc+4`, `cpu.pc = (fw_a+imm)&~1`, `flush=true` | `reg_write=true` |
| `0x37` (LUI) | `alu_result = imm` | `reg_write=true` |
| `0x17` (AUIPC) | `alu_result = pc + imm` | `reg_write=true` |
| `0x73` (SYSTEM) | ecall/ebreak → `cpu_trap`；mret → 恢复 priv+pc；`flush=true` | — |
| `0x0F` (FENCE) | NOP | — |
| `0x07` (FLW) | `alu_result = fw_a + imm` | `is_fp=true`, `mem_read=true` |
| `0x27` (FSW) | `alu_result = fw_a + imm`, `rs2_val = fregs[rs2]` | `is_fp=true`, `mem_write=true` |
| `0x53` (OP-FP) | `exec_fp_op` → 写 `fregs[rd]` | `is_fp=true` |
| `0x43/47/4B/4F` (FMA) | `exec_fma` → 写 `fregs[rd]` | `is_fp=true` |

### 3.10 流水线填充与排空

流水线通过各寄存器的 `valid` 标志自然地填充和排空：

```
Cycle   IF/ID    ID/EX    EX/MEM   MEM/WB   指令计数
  1    ✓ valid  ✗ invalid ✗ invalid ✗ invalid    0
  2    ✓ valid  ✓ valid  ✗ invalid ✗ invalid    0
  3    ✓ valid  ✓ valid  ✓ valid  ✗ invalid    0
  4    ✓ valid  ✓ valid  ✓ valid  ✓ valid      0
  5    ✓ valid  ✓ valid  ✓ valid  ✓ valid      1  ← 第一条 WB
  6    ✓ valid  ✓ valid  ✓ valid  ✓ valid      2  ← 稳态 (1 IPC)
  ...
```

以 3 条指令的测试程序（addi + addi + ecall）为例：

- 周期 1–4：流水线填充（4 周期）
- 周期 5：第一条 addi 完成（WB）
- 周期 6：第二条 addi 完成（WB）
- 周期 7：ecall 完成（WB）

> **总计 7 周期**（3 条指令：5 周期填充 + 2 额外指令 = 7）

---

## 4. 三种模型对比

| 指标 | 单周期 | 多周期 | 流水线 |
|------|:------:|:------:|:------:|
| CPI（minimal.elf, 3 条指令） | 1.00 | 3.67 | 2.33 |
| 周期数（3 条指令） | 3 | 11 | 7 |
| 最大同时飞行指令数 | 1 | 1 | 5 |
| Forwarding | — | — | EX/MEM + MEM/WB → EX |
| Stall (load-use) | — | — | 1 周期 |
| Flush (branch) | — | — | 2 周期 |
| 每周期工作量 | 1 条完整指令 | 1 个 FSM 状态 | 5 级并行 |
| 实现复杂度 | 低 | 中 | 高 |

## 5. 命令行用法

```bash
# 单周期（默认）
./riscv-sim hello.elf

# 多周期
./riscv-sim -m multi hello.elf

# 流水线
./riscv-sim -m pipeline hello.elf
```

`-m` 参数由 `main.c` 中的 `parse_cpu_model()` 解析：

```c
static CpuModel parse_cpu_model(const char *s) {
    if (strcmp(s, "multi")    == 0) return MODEL_MULTI_CYCLE;
    if (strcmp(s, "pipeline") == 0) return MODEL_PIPELINE;
    if (strcmp(s, "single")   == 0) return MODEL_SINGLE_CYCLE;
    return MODEL_SINGLE_CYCLE;  // 未知 → 默认
}
```

## 6. 测试结果

三种模型产生**完全一致的架构状态**（相同寄存器值、相同指令数、相同异常码），但周期数不同：

```
Single-cycle:  3 instructions,  3 cycles  (CPI = 1.00)
Multi-cycle:   3 instructions, 11 cycles  (CPI = 3.67)
Pipeline:      3 instructions,  7 cycles  (CPI = 2.33)
```

## 7. 局限与未来工作

### 当前局限

| 问题 | 说明 |
|------|------|
| **FP 转发** | FP 寄存器文件无转发路径。相邻 FP 指令有 RAW 冒险时会读到过期数据。（FP 指令在典型 RISC-V 程序中较少，当前测试未涉及。） |
| **分支预测** | 始终预测不跳转。简单的 1-bit 或 2-bit 动态预测器可减少误预测惩罚。 |
| **精确异常** | EX 阶段陷阱发生后，后续指令被冲刷，但陷阱指令的 PC 并非在所有情况下都精确（例如 MEM 阶段 Load 故障时）。 |
| **CSR 指令** | 仅实现 ECALL/EBREAK/MRET。完整的 CSR 读写指令（CSRRW, CSRRS, CSRRC 等）未实现，触发非法指令异常。 |

### 未来方向

- 超标量（2-way / 4-way 发射）
- 乱序执行 + 寄存器重命名
- 分支预测器（BTB + BHT + RAS）
- 非阻塞缓存
- FP 寄存器转发
