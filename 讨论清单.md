# 团队讨论清单 —— 需要共同对齐的决策

> 在各自深入编码之前，以下事项需要四个人一起讨论并拍板。每条都标注了**影响谁**和**不拍板的后果**。

---

## 🔴 最高优先级：数据结构 & 内存接口（阻塞所有人）

### 1. 内存模块的架构：一层还是两层？

两个参考项目走了不同的路，必须选一个：

| 方案 | 参考 | 结构 | 特点 |
|------|------|------|------|
| **A：两层**（物理内存 + MMU） | 参考项目1 | `PhysicalMemory`（字节数组 + Region 列表）+ `MMUState`（页表、satp、地址翻译）| 接近真实硬件，职责清晰 |
| **B：单层**（虚拟内存即内存） | 参考项目2 | `MMU` 直接维护 `MemRegion[]`（虚拟地址区间 + 数据缓冲区），没有物理内存概念 | 简单，够用。不需要真正模拟物理页帧 |

**影响谁**：焕聪（Memory 的实现方式完全不同）、李特（调用的接口不同）、嘉华（Loader 写入方式不同）

**李特文档里假设的是方案 B**（`mmu_read32(mmu, addr, val)` 直接操作虚拟地址），参考项目 1 的接口是两层的（`mmu_read_32(mmu, pmem, vaddr, val, priv)`）。

> ⚠️ **不拍板的后果**：焕聪按一层写，Loader 发现需要物理页分配能力，返工。或者按两层写，CPU 发现每次访存要多传一个 `pmem` 参数。

### 2. Memory 模块对外暴露什么接口？（函数签名）

这是整个项目最重要的接口——CPU、Loader、Debugger 都通过它访问内存。必须精确对齐：

```
当前 李特文档 里假设的接口：
  int mmu_read32(MMU*, uint32_t addr, uint32_t *val);
  int mmu_read16(MMU*, uint32_t addr, uint32_t *val);
  int mmu_read8(MMU*, uint32_t addr, uint32_t *val);
  int mmu_write32(MMU*, uint32_t addr, uint32_t val);
  int mmu_write16(MMU*, uint32_t addr, uint32_t val);
  int mmu_write8(MMU*, uint32_t addr, uint32_t val);
  int mmu_read(MMU*, uint32_t addr, uint8_t *buf, uint32_t len);    // 批量读
  int mmu_write(MMU*, uint32_t addr, const uint8_t *buf, uint32_t len); // 批量写
  int mem_map(MMU*, uint32_t base, uint32_t size, uint8_t prot);    // Loader 分配区域

参考项目1 的接口（两层）：
  bool mem_read_32(PhysicalMemory*, uint32_t addr, uint32_t *val);
  bool mmu_read_32(MMUState*, PhysicalMemory*, uint32_t vaddr, uint32_t *val, PrivilegeLevel);
  bool mmu_map_page(MMUState*, uint32_t vaddr, uint32_t paddr, uint8_t flags);
```

需要拍板的内容：
- [ ] **返回值用什么**：`int`（0 成功 / 非 0 错误码）还是 `bool`（true 成功）？
- [ ] **函数名前缀**：`mmu_` 还是 `mem_`？
- [ ] **地址翻译在哪里做**：MMU 函数内部自动翻译，还是调用者先翻译再传物理地址？
- [ ] **异常怎么报**：通过返回值？还是设置 `ExceptionType*` 输出参数？
- [ ] **批量读写**（`mmu_read`/`mmu_write`）是否需要？syscall write 需要从内存批量读数据

**影响谁**：焕聪（实现者）、李特（每天调几十次）、嘉华（Loader 段加载 + brk）、嘉俊（`x` 命令读内存）

> ⚠️ **不拍板的后果**：四个人各自假设一套接口，联调时发现函数名/参数/返回值全对不上，每个人的代码都要改。

### 3. Simulator 顶层结构体

这是整个模拟器的"大总管"，主循环 `main.c` 需要它来串起所有模块。必须一起定下字段：

```c
typedef struct {
    CPU         cpu;           // CPU 状态         → 李特负责定义
    MMU         mmu;           // 内存管理          → 焕聪负责定义
    Breakpoint  *breakpoints;  // 断点数组          → 归谁？
    int         bp_count;      // 断点数量
    int         bp_capacity;   // 断点数组容量
    bool        single_step;   // 单步标志
    bool        debug_mode;    // 调试模式标志
    uint64_t    inst_count;    // 指令计数
    uint64_t    cycle_count;   // 周期计数
} Simulator;
```

需要拍板的内容：
- [ ] **断点数组放在哪里**？Simulator 下面（参考项目2）还是 Debugger 内部（参考项目1）？
  - 放 Simulator → CPU 执行 `ebreak` 时可以直接查 `sim->breakpoints[]`
  - 放 Debugger → CPU 需要知道 Debugger 的指针，耦合更深
- [ ] **CSR 寄存器放哪里**？李特的 CPU 只有 `regs[32] + pc + running`，没有 CSR。
  - 参考项目1 的 CPUState 包含 mtvec / mepc / mcause / mstatus 等
  - 没有 CSR 就无法处理异常（段错误时 mepc 写哪里？）
- [ ] **`single_step` 标志谁写谁读**？Debugger 写、CPU 主循环读

**影响谁**：所有人（main.c 是共同的地基）

> ⚠️ **不拍板的后果**：四个人模块写好了，发现拼不到一起——结构体字段名对不上、断点放在 Debugger 里但 CPU 访问不到。

### 4. CPU 状态结构体需要包含什么？

对比两个版本：

| 字段 | 李特版 | 参考项目1 版 |
|------|--------|-------------|
| 通用寄存器 regs[32] | ✅ | ✅ |
| PC | ✅ | ✅ |
| running | ✅ | ✅ |
| next_pc | ❌ | ❌（独立变量） |
| 特权级 priv | ❌ | ✅ |
| mtvec | ❌ | ✅ |
| mepc | ❌ | ✅ |
| mcause | ❌ | ✅ |
| mtval | ❌ | ✅ |
| mstatus | ❌ | ✅ |
| mie / mip | ❌ | ✅ |
| mscratch | ❌ | ✅ |

CSR 寄存器看起来"没用"，但以下场景需要它们：
- Load/Store 地址未对齐或越权 → 触发异常 → 需要写 `mcause` + `mtval`，跳转到 `mtvec`
- ECALL → 需要读 `mtvec` 找到陷阱处理程序入口

**如果你们决定异常处理从简**（遇到异常直接 `printf` + 退出），CSR 不需要。但如果要正确模拟 RISC-V 异常机制，CSR 必须在 CPU 结构体里。

**影响谁**：李特（CPU 结构体）、焕聪（地址翻译失败后怎么通知 CPU）

## 🟡 高优先级：模块边界 & 交互

### 5. 系统调用（syscall）归谁实现？

这是个关键的边界问题，参考项目里两种做法：

| 方案 | 做法 | 参考 |
|------|------|------|
| **A：独立 syscall 模块** | `syscall.c` 文件，`syscall_handler()` 函数 | 参考项目1 |
| **B：放在 CPU 执行阶段** | ECALL 的 switch-case 直接写 syscall 处理 | 李特文档 §3.4 时序2 |

方案 B 的代码在李特文档里已经勾勒出来了（判断 a7 的值，调 mmu_read/write）。但 syscall 涉及 stdin/stdout 交互、brk 管理（需要 Memory 支持），不完全属于 CPU 的"计算"范畴。

需要拍板：
- [ ] syscall 是**独立的 `.c` 文件**（焕聪或嘉华负责？），还是李特在 `execute.c` 里顺手写了？
- [ ] `SYS_brk` 需要 Memory 提供什么接口？（`mmu_brk` / `mmu_extend_heap`?）这个接口需要焕聪先定义好

**影响谁**：李特（ECALL 执行时会调用 syscall 逻辑）、焕聪（brk 需要 Memory 支持）、嘉华（Loader 初始 brk 地址）

### 6. 断点的实现方式：ebreak 替换还是地址比对？

李特文档 §3.4 时序3 描述了"ebreak 替换"方案：
```
设置断点时：Debugger 把目标地址的原指令保存，替换为 ebreak (0x00100073)
CPU 执行到 ebreak → 暂停 → Debugger 恢复原指令，PC 回退一条
继续执行时：Debugger 单步执行原指令，然后重新插入 ebreak
```

这是参考项目1的做法。但还有另一种"地址比对"方案（不修改内存，CPU 每执行一条指令前对比 PC 和断点列表）。

需要拍板：
- [ ] **用哪种方案**？
  - ebreak 替换：经典做法，但需要 Debugger 能写内存（通过 `mmu_write32` 替换指令），CPU 需要知道断点列表
  - 地址比对：简单但对每条指令都有开销
- [ ] **CPU `ebreak` 处理时怎么区分"这是断点"还是"程序自己的 ebreak"**？（查断点列表）
- [ ] **断点列表是 CPU 可见还是 Debugger 独占**？

**影响谁**：李特（ebreak 处理逻辑）、嘉俊（断点管理 + 指令替换）

### 7. Loader → CPU 的启动参数传递

ELF 加载完成后，需要告诉 CPU：
- **入口地址**（`elf_header.e_entry`）→ 设 `cpu.pc`
- **栈指针** → 设 `cpu.regs[SP]`

需要拍板：
- [ ] **栈地址是多少**？参考项目1 用了 `0x7FFFF000`（高位栈），李特文档用了 `0xC0000000`（参考项目2）
  - `0x7FFFF000`：2GB 位置，给代码/数据留了巨大空间
  - `0xC0000000`：3GB 位置，更接近真实 Linux 用户空间布局
- [ ] **栈大小**？64KB（0x10000）还是 1MB（0x100000）？
- [ ] **Loader 还需要设什么**？`regs[GP]`？`regs[TP]`？argv/envp 要不要传？

**影响谁**：嘉华（Loader 设置这些值）、李特（CPU 依赖这些初值）

## 📝 讨论结论记录区

> 以下结论于 2026-07-01 确定。详细接口规范见 `docs/设计文档/共同部分/公共类型定义.md`。

### 结论 1：内存架构 ✅
- **选型：方案 A — 两层架构**（PhysicalMemory + MMUState）
- **原因**：
  1. 贴近真实硬件，职责分离清晰：PhysicalMemory 管字节数组 + 边界检查，MMU 管地址翻译 + 权限校验
  2. 各模块各司其职——Loader 调 PhysicalMemory 层加载数据 + 调 MMU 层建立映射，CPU 只调 MMU 层，Debugger 可穿透到 PhysicalMemory 层 dump
  3. 保留扩展空间：后续可加 TLB、缺页中断、大页支持
  4. Bare 模式（satp=0）下 vaddr=paddr，行为上等价于单层，不增加无谓开销
- **对 CPU 的影响**：CPU 调用 `mmu_read_32(mmu, pmem, vaddr, &val, priv, &exc)`，需要传 `MMUState*` 和 `PhysicalMemory*` 两个指针（都从 `Simulator*` 取）

### 结论 2：Memory 接口规范 ✅
- **返回值类型**：`bool` — `true` 成功，`false` 失败
- **错误详情**：通过 `ExceptionType* exc` 输出参数传递（调用者传入 `&exc`）
- **函数名前缀**：`mem_` 用于 PhysicalMemory 层，`mmu_` 用于 MMU 层
- **地址翻译位置**：在 MMU 函数内部自动完成（CPU 传虚拟地址，MMU 翻译后调 PhysicalMemory）
- **参数顺序**：`(自身状态, 被操作对象, 地址, 数据, 权限/异常)` — 自身状态指针始终是第一个参数
- **批量读写保留**：`mmu_read()` / `mmu_write()` 用于 syscall 场景
- **CPU 只调 MMU 层，不直接调 `mem_*`**；Loader 调 `mem_map` + `mem_load` + `mmu_map_page`

### 结论 3：Simulator 结构体 ✅
```c
typedef struct {
    CPU             cpu;           // CPU 状态           → 李特
    MMUState        mmu;           // MMU 页表/satp      → 焕聪
    PhysicalMemory  pmem;          // 物理内存            → 焕聪
    Breakpoint     *breakpoints;   // 断点数组（放 Simulator 下）
    int             bp_count;
    int             bp_capacity;
    bool            single_step;   // 单步标志
    bool            debug_mode;    // 调试模式标志
    uint64_t        inst_count;    // 指令计数
    uint64_t        cycle_count;   // 周期计数
} Simulator;
```
- **断点放 Simulator 下**（而非 Debugger 内部）。原因：CPU 执行 ebreak 时需要查断点列表区分"断点触发"还是"程序自身的 ebreak"；CPU 不能反向依赖 Debugger

### 结论 4：CPU 结构体 — 加 CSR ✅
- **决定：必须包含最小 CSR 集合**
- CPU 结构体最终字段：
  ```c
  typedef struct {
      uint32_t regs[32];       // 通用寄存器
      uint32_t pc;
      bool     running;
      PrivilegeLevel priv;     // 当前特权级
      uint32_t mstatus;        // MIE | MPIE | MPP
      uint32_t mtvec;          // 陷阱向量基址
      uint32_t mepc;           // 异常返回地址
      uint32_t mcause;         // 异常原因
      uint32_t mtval;          // 异常相关信息
  } CPU;
  ```
- **原因**：Load/Store 访存失败时需要写 `mcause` + `mtval` + `mepc` 并跳转到 `mtvec`；即使本阶段简化为 printf+exit，CSR 字段也作为结构体的一部分保留
- **暂不包含**：`mie`/`mip`（中断相关，单核无外设不需要）、`mscratch`（非必需）

### 结论 5：syscall 归属 ✅
- **独立 `syscall.c` 模块**（不放在 CPU 的 execute.c 中）
- **原因**：
  1. syscall 涉及 stdin/stdout 交互、brk 管理（需调 PhysicalMemory），不完全属于 CPU 的"计算"范畴
  2. 独立模块便于焕聪维护 brk 相关逻辑
  3. CPU 的 ecall 处理只做一件事：调用 `syscall_handler(sim)`
- **syscall_handler 签名**：`bool syscall_handler(Simulator *sim);`

### 结论 6：断点方案 ✅
- **采用 ebreak 替换方案**（参考项目1 做法）
- 设置断点时：Debugger 保存原指令 → 用 `mmu_write_32` 替换为 `ebreak` (`0x00100073`)
- 触发断点时：CPU 执行到 ebreak → 查 `sim->breakpoints[]` 确认 → 恢复原指令，PC 回退
- 继续执行时：Debugger 单步执行原指令，然后重新插入 ebreak
- **断点列表位于 Simulator 下**，CPU 和 Debugger 都可访问

### 结论 7：Loader → CPU 参数 ✅
- **栈地址**：`0xC0000000`（栈顶），和参考项目2 一致
- **栈大小**：1MB（`0x100000`），从 `0xBFF00000` 到 `0xC0000000`
- **Loader 设置的值**：
  - `sim->cpu.pc = elf_entry`（ELF 入口地址）
  - `sim->cpu.regs[2] = 0xC0000000`（sp，栈顶）
  - 其他寄存器保持 0，由程序自己初始化 gp 等

### 结论 8：构建系统 ✅
- 先用 **Makefile**（README 里已写 `make`）
- 编译器和标志：`gcc -std=c11 -Wall -Wextra -g -O0`
- include 路径：`-Isrc/include`
- 后续如需跨平台可迁移到 CMake

### 结论 9：文件组织 ✅
- 头文件按模块分目录：`src/include/{cpu,memory,loader,debugger}/`
- 公共头文件 `types.h` 放在 `src/include/` 根目录
- `#include` 写法：`#include "types.h"` 或 `#include "memory/memory.h"`，编译时加 `-Isrc/include`
- 不使用 `common.h` 这种万能头文件，各模块精确包含自己需要的头文件

### 结论 10：测试环境 ✅
- 嘉华负责搭建 RISC-V 交叉编译环境（`riscv64-unknown-elf-gcc`）
- 测试 `.s` 汇编文件和 `.elf` 放在 `tests/` 目录
- Windows 上推荐使用 WSL2 或预编译的 RISC-V toolchain
- 最简单的验证：写一个 `addi a0, zero, 42; ecall` 的汇编，期望 a0=42 退出
